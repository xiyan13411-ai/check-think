# 愿望拼图存钱 App 完整规划路线

> Repository: `xiyan13411-ai/check-think`  
> 建议路径：`docs/ROADMAP.md`  
> 路线原则：先跑通爽感闭环，再做平台化和双端扩展。

---

## 0. 总路线概览

项目建议分为 7 个阶段：

1. Phase 0：项目基线确认
2. Phase 1：MVP 静态骨架
3. Phase 2：存钱逻辑与本地数据
4. Phase 3：拼图碎片与动效爽感
5. Phase 4：历史记录、文案与成就
6. Phase 5：PWA 与安卓体验优化
7. Phase 6：Capacitor 打包与双端预留
8. Phase 7：增强版功能与产品化

每个阶段都要保持一个原则：

> 每一步都应该能运行、能体验、能回退。

不要让 Codex 一次性开发巨大功能包。大任务拆小，逐步验收。

## Phase 1：MVP 静态骨架

### 页面内容

## Phase 2：存钱逻辑与本地数据

### 目标

实现存钱闭环和 localStorage 持久化。

### 需要创建的类型

```ts
Goal
SavingRecord
Achievement
AppState
```

### 需要创建的工具

- `lib/storage.ts`
- `lib/progress.ts`
- `lib/copywriting.ts`
- `lib/vibration.ts`

### 核心逻辑

1. 初始化默认目标。
2. 从 localStorage 读取状态。
3. 如果没有状态，则创建默认状态。
4. 点击快捷金额增加当前金额。
5. 输入自定义金额后增加当前金额。
6. 计算进度百分比。
7. 计算解锁碎片数量。
8. 每次存钱生成一条记录。
9. 保存状态到 localStorage。

### 验收标准

- 点击 ¥10 / ¥50 / ¥100 / ¥500 都能增加金额。
- 自定义金额能正常提交。
- 非法金额不能提交。
- 当前金额不能变成 NaN。
- 刷新页面后金额和记录不丢。
- TypeScript 无明显类型错误。

## Phase 3：拼图碎片与动效爽感

### 目标

让用户“看见愿望拼出来”。

### 拼图规则

- 默认 40 个碎片。
- 根据进度决定已解锁碎片数。
- 已解锁碎片高亮显示。
- 未解锁碎片半透明或灰色。
- 新增碎片播放飞入动画。

### 动画要求

使用 Framer Motion：

1. 新碎片从屏幕边缘或下方飞入。
2. 按钮点击时 scale。
3. 金额变化时轻微高亮。
4. 拼图区域在解锁时轻微发光。

### 性能要求

- 仅使用 2D 动效。
- 不引入 Three.js。
- 不做复杂图片处理。
- 动画尽量基于 `transform` 和 `opacity`。

### 验收标准

- 存钱后拼图碎片数量增加。
- 新碎片有明显飞入动画。
- 动画不卡成 PPT。
- 即使没有真实图片，视觉上也像“目标正在拼出来”。

## Phase 4：历史记录、文案与成就

### 目标

补齐情绪价值和持续反馈。

### 历史记录

每次存钱生成一张小票式记录：

- 金额
- 时间
- 解锁碎片数
- 进度
- 随机文案

首页默认展示最近 5 条。

### 文案库

示例：

- 叮！未来的愿望回来了一小块。
- 这不是一笔钱，是一块自由。
- 存钱这事，终于有画面了。
- 今天也偷偷变有钱了。
- 小金库 +1，焦虑 -1。
- 愿望不是突然实现的，是这样一块一块拼回来的。

### 成就

节点：

- 25%
- 50%
- 75%
- 100%

成就只在首次达到时弹出。

### 验收标准

- 存钱后出现随机文案。
- 历史记录新增。
- 历史记录刷新后不丢。
- 成就弹窗能出现。
- 成就不会重复弹出。
- 文案轻盈，不油腻。

## Phase 5：PWA 与安卓体验优化

### 目标

让网页更像手机 App。

### PWA 文件

创建：

- `app/manifest.ts`
- `/public/icons/icon-192.png`
- `/public/icons/icon-512.png`

如果图标暂时没有，可以先用占位图或在 README 说明。

### Manifest 建议

```ts
export default function manifest() {
  return {
    name: "愿望拼图存钱",
    short_name: "存钱拼图",
    description: "每存一笔钱，愿望就拼回来一块",
    start_url: "/",
    display: "standalone",
    background_color: "#fff7ed",
    theme_color: "#f9a8d4",
    icons: [
      {
        src: "/icons/icon-192.png",
        sizes: "192x192",
        type: "image/png"
      },
      {
        src: "/icons/icon-512.png",
        sizes: "512x512",
        type: "image/png"
      }
    ]
  };
}
```

### 安卓优化

- 数字输入键盘。
- 触摸区域足够大。
- 成功存钱后震动。
- safe area 预留。
- 页面不横向滚动。
- 弱网时尽量不白屏。
- 动画不重。

### 验收标准

- Android Chrome 中页面正常。
- 可添加到主屏幕的基础配置存在。
- 页面像 App，不像普通网页。
- README 说明 PWA 使用方式。

## Phase 6：Capacitor 打包与双端预留

### 目标

在 Web MVP 稳定后，把项目包装成 Android App，并预留 iOS。

### 触发条件

只有当 Phase 1 - Phase 5 都稳定后再做。

### 任务

1. 添加 Capacitor。
2. 配置 Android 项目。
3. 确认 Next.js 是否可静态导出。
4. 配置打包目录。
5. 生成 Android 测试包。
6. 记录打包步骤。
7. 预留 iOS 配置说明。

### 注意

- 不要在 MVP 初期过早引入 Capacitor。
- 先验证“存钱爽感”再上原生壳。
- 如果项目依赖 SSR，需要先调整为静态页面或重新规划打包方式。

### 验收标准

- 能生成 Android 测试包。
- 安装后首页可打开。
- 存钱功能正常。
- 本地数据正常。
- 文档中有打包说明。

## MVP 完成定义

当以下条件全部满足，可以认为 MVP 完成：

- 用户能在手机浏览器打开页面。
- 用户能看到默认愿望目标。
- 用户能点击存钱。
- 存钱后金额、进度、碎片都会变化。
- 存钱后有动画和文案。
- 存钱后有记录。
- 刷新后数据不丢。
- 达成节点会弹成就。
- 页面看起来可爱，不像后台。
- 没有接入支付、银行、后端。
- README 能说明如何运行。

---

## 产品化前的最后检查

正式扩展前，先问 5 个问题：

1. 点击存钱时，有没有爽感？
2. 拼图变化是否一眼能看出来？
3. 用户是否愿意第二天再打开？
4. 页面是否适合截图分享？
5. 继续做多目标、云同步、App 打包是否值得？

如果前 3 个答案不明确，不要急着做复杂功能。先把核心体验磨顺。
