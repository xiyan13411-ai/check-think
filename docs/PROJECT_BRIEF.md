## 1. 项目概述

「愿望拼图存钱 App」是一个情绪价值型存钱工具。它不直接接入真实支付、银行账户或理财产品，而是让用户手动记录每一笔存下来的钱，并通过拼图、动画、文案和成就反馈，让“存钱”变得像“收集愿望碎片”一样有爽感。

核心体验是：

> 用户创建一个愿望目标，例如新手机、旅行、相机、礼物或自由基金。  
> 每存一笔钱，目标物品就解锁一部分碎片。  
> 随着存钱进度提升，愿望从灰色轮廓逐渐拼成完整物品。

这个项目的重点不是复杂财务功能，而是“可视化反馈 + 情绪奖励 + 移动端上手快”。

---

## 2. 项目背景

传统存钱、记账工具通常只展示金额、账单和进度条，反馈偏理性；而花钱行为往往有立刻的商品、快递、包装、截图和社交反馈，所以用户更容易对“消费”产生即时满足。

本项目希望反过来设计一种体验：

- 花钱有快感，存钱也应该有快感。
- 存钱不是冷冰冰的数字，而是愿望一点点被拼出来。
- 用户每次点击“存入 ¥10 / ¥50 / ¥100”，都能获得即时反馈。
- 让用户从“我少花了钱”转变为“我拼回了一块未来”。

## 4. 目标用户

### 4.1 核心用户

1. 想为具体愿望存钱的人  
   例如手机、相机、旅行、搬家、礼物、兴趣消费。

2. 对传统记账 App 无感的人  
   不喜欢复杂分类、预算图表和资产负债表，只想轻松记录。

3. 喜欢可爱、轻量、即时反馈的人  
   能接受游戏化、养成感、碎片解锁、成就徽章。

4. 安卓手机用户  
   当前第一优先级为安卓手机体验，后续扩展 iOS。

### 4.2 使用场景

- 今天少买了一杯奶茶，记录“存下 ¥20”。
- 每周固定存 ¥100，看目标物品逐渐拼完整。
- 想买手机，于是每天拼一点“新手机碎片”。
- 睡前打开看一眼进度，获得一点“我在变好”的反馈。
- 达成目标后截图分享给朋友或伴侣。

## 5. 产品目标

### 5.1 MVP 目标

MVP 阶段要完成一个可运行、可演示、移动端好看的闭环版本。

必须实现：

1. 展示默认愿望目标。
2. 支持快捷金额存钱。
3. 支持自定义金额存钱。
4. 计算当前金额、目标金额、进度百分比。
5. 根据进度显示拼图碎片。
6. 每次存钱后播放碎片飞入动画。
7. 每次存钱后显示随机鼓励文案。
8. 每次存钱生成历史记录。
9. 数据保存在 localStorage，刷新不丢。
10. 安卓手机浏览器体验优先。

### 5.2 非目标

MVP 暂不实现：

- 登录注册
- 云同步
- 后端服务
- 数据库
- 真实支付
- 银行账户绑定
- 投资理财建议
- 自动记账
- 多人协作
- 社交关系链
- 复杂 3D 动画
- App Store / Google Play 上架

## 6. 核心玩法

### 6.1 愿望目标

用户围绕一个愿望目标进行存钱。

MVP 默认目标：

- 目标名称：新手机基金
- 目标金额：10000
- 当前金额：0
- 货币符号：¥
- 拼图碎片数：40

后续可支持：

- 多个目标
- 自定义目标图片
- 自定义目标金额
- 截止日期
- 主题颜色
- 目标分类

### 6.2 存钱动作

用户可以通过以下方式记录存钱：

- 快捷金额：¥10
- 快捷金额：¥50
- 快捷金额：¥100
- 快捷金额：¥500
- 自定义金额输入

成功存钱后发生：

1. 当前金额增加。
2. 进度百分比更新。
3. 新解锁碎片从屏幕边缘飞入。
4. 金额区域出现轻微高亮或放大反馈。
5. 随机显示一条鼓励文案。
6. 生成小票式历史记录。
7. 安卓浏览器支持时触发轻微震动。

### 6.3 拼图解锁

核心计算：

```ts
progress = Math.min(currentAmount / targetAmount, 1)
unlockedPieces = Math.floor(progress * totalPieces)
```

例如：

- 目标金额 10000
- 总碎片 40
- 当前金额 2500
- 进度 25%
- 解锁碎片 10 块

MVP 不需要真实切图，可以使用 CSS Grid 模拟 40 个碎片。目标物品可以先用抽象手机卡片、emoji、SVG 或渐变块来表示。

### 6.4 成就节点

当进度首次达到以下节点时，弹出成就：

| 节点 | 成就名 | 文案 |
|---|---|---|
| 25% | 愿望长出边角 | 已经拼出 25%，有点东西了。 |
| 50% | 已经拼出一半 | 这不是进度条，是底气条。 |
| 75% | 快要摸到了 | 离目标很近了，别停。 |
| 100% | 目标达成 | 愿望拼完整了，漂亮。 |

## 8. UI 设计要求

### 8.1 总体风格

- 奶油白背景
- 浅粉、浅黄、浅橙点缀
- 大圆角卡片
- 柔和阴影
- 少量玻璃拟态
- 小红书感
- 不要后台系统感
- 不要金融 App 的严肃感

### 8.2 移动端规则

- 优先适配 360px - 430px 宽度。
- 禁止横向滚动。
- 按钮高度不低于 44px。
- 底部操作区域留出安全区。
- 自定义金额输入使用数字键盘。
- 单手操作舒适。
- 页面不要过长，核心操作尽量在首屏或一屏半内完成。

### 8.3 视觉层级

金额数字最大，其次是目标名称，再其次是鼓励文案和记录。

建议层级：

1. 当前金额
2. 拼图主体
3. 快捷存钱按钮
4. 进度百分比
5. 历史记录

## 9. 动效设计

### 9.1 必要动效

使用 Framer Motion 实现：

1. 按钮点击缩放。
2. 新碎片飞入。
3. 历史记录淡入上移。
4. 成就弹窗弹出。
5. 金额区域轻微高亮。

### 9.2 动效原则

- 快，不拖沓。
- 轻，不影响操作。
- 明确，让用户知道“我刚刚做了什么”。
- 节制，不要每个元素都乱飞。
- 安卓中低端机也要尽量顺滑。

### 9.3 震动反馈

封装设备震动函数：

```ts
export function vibrate(duration = 30) {
  if (typeof navigator !== "undefined" && "vibrate" in navigator) {
    navigator.vibrate(duration);
  }
}
```

仅在成功存钱后调用。

## 13. 风险与解决方案

### 风险 1：动画复杂导致开发卡住

解决：MVP 只做 2D CSS Grid 碎片和 Framer Motion，不做 3D。

### 风险 2：功能变成普通记账

解决：核心始终围绕“愿望拼图”而不是账本分析。

### 风险 3：本地数据丢失

解决：MVP 明确使用本地存储；后续增加导出和云同步。

### 风险 4：后续双端重构成本高

解决：从第一版开始封装存储、震动、平台能力和业务类型。

### 风险 5：金融合规问题

解决：不接银行、不做支付、不推荐理财、不承诺收益，只做手动记录。
